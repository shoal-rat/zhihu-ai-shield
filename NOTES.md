# 设计笔记

## 从描述到实现

原始想法是：刷知乎时把正在加载内容的作者拉出来，抓他近几篇内容，过一遍模型；如果明显低质量，就自动屏蔽，并把页面内容改写成“AI 已挡下”的状态。

当前版本对应实现如下：

- 内容脚本监听知乎页面动态加载
- 对每个内容卡片提取作者、标题、正文摘要
- 后台脚本按作者去重，避免同一作者反复请求模型
- 尝试读取作者主页、回答页、文章页、想法页
- 本机 LLM 返回 JSON 决策，扩展按阈值和置信度执行
- 命中后在页面本地改写、折叠、模糊或隐藏
- 用户可以临时展开，也可以把作者加入白名单

## 为什么不直接调用知乎拉黑

这个仓库暂时不内置官方拉黑请求：

- 站内接口容易变化
- 自动账号操作风险更高
- 误判后的恢复成本更高
- 本地屏蔽足够覆盖阅读流场景

如果未来要做，也应该是显式开关、二次确认、可审计日志，而不是默认全自动。

## 为什么要缓存

模型调用是最慢、最贵、也最容易产生不稳定结果的一步。扩展按作者缓存判断结果，默认 168 小时，能减少重复请求，也能避免页面滚动时卡顿。

## 为什么只用本机 LLM

这个版本不再做关键词兜底评分。模型不可用、模型返回不合法或本机服务未启动时，扩展只记录错误并放行当前内容，避免用硬编码词表误伤。

## 游客兼容边界

扩展会在未登录时继续处理页面上已经公开加载出来的内容，但不会伪造登录态、不会调用绕过登录的第三方代码，也不会试图读取服务端没有返回的内容。
